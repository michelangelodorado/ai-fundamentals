1 CPU vs GPU

Why AI runs on GPUs at all.

A CPU has a few very powerful cores built for sequential logic. A GPU has thousands of simple cores that do the same math on lots of data at once. LLMs are giant matrix multiplications — embarrassingly parallel — so GPUs crush them.

2 CUDA & the NVIDIA Moat

Why NVIDIA is so hard to displace.

CUDA is NVIDIA's platform for programming GPUs. Every major AI framework — PyTorch and the rest — is built on it. Even when a competitor's chip is fast, the entire ecosystem is written for CUDA, so switching is painful. That's the moat.

3 VRAM — the Constraint ★ the big one

The single most important spec for inference.

The whole model's weights + the KV cache (Section 2) must fit in GPU memory. A 70B model at FP16 needs ≈140 GB — more than one 80 GB GPU. So you either quantize it smaller or split it across GPUs.

Your world: "how many GPUs do I need?" is really "how much VRAM does the model + expected KV cache need?" — the first sizing question in any AI infra deal.

4 Tensor vs Pipeline Parallelism

Two ways to split a model that won't fit on one GPU.

Tensor parallelism slices each layer across GPUs — they combine results constantly, so they need fast links (best inside one server). Pipeline parallelism splits by layers like an assembly line — less chatty, so it works across servers.

5 NVLink vs PCIe (and NCCL)

How GPUs talk — and why it matters for splitting.

NVLink is NVIDIA's ultra-fast direct link between GPUs inside a server. PCIe is the slower standard bus to the CPU. Tensor parallelism chats constantly, so it needs NVLink. NCCL is the software that moves the data efficiently over whichever link.

6 DPUs

Offloading everything that isn't compute.

A DPU (Data Processing Unit, e.g. NVIDIA BlueField) is a smart NIC that takes networking, storage, and security off the CPU — so the CPU and GPUs can focus on the model. In AI datacenters, this is where traffic management, encryption, and security enforcement increasingly live.

7 MIG & GPU Scheduling

The opposite problem: when one GPU is too big for one small job.

MIG (Multi-Instance GPU) partitions a single GPU into up to 7 isolated instances, so several small workloads share it safely instead of one job hogging it. Scheduling — usually via the Kubernetes GPU Operator — decides who gets which slice when. Utilization is money.

8 Inference Servers & Batching

A model file doesn't serve traffic — an inference server does.

It loads the model onto GPUs and exposes an API, handling batching (combining concurrent requests so the GPU stays busy), KV-cache management, and streaming. Batching is the key throughput trick.
