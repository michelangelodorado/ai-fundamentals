1 Zero / One / Few-shot

"Shots" = how many examples you put in the prompt before the real task.

Zero-shot just asks. One-shot shows one example. Few-shot shows several — the model locks onto your format and edge cases. More examples → more consistency, but they cost tokens.

2 Role & Structured Prompting

    
Two cheap, high-impact habits.

    
Role prompting assigns a persona to set depth and tone. Structured prompting organizes the request into clear sections — the single biggest quality lever most people skip.

3 XML Prompting

    
Tags tell the model exactly where data ends and instructions begin.

    
When you paste a document and give instructions, how does the model know which is which? Wrapping each part in tags removes the ambiguity — and it's the first line of defense against instructions hidden inside your data (more in §9).

4 JSON Outputs

    
Turn the model from a chatbot into a software component.

    
Ask for JSON and your app can parse the answer directly instead of scraping prose. This is what lets an LLM slot into a pipeline — and it pairs with tool calling from Section 2.

5 Prompt Chaining

    
Break a big task into a sequence of simple, checkable steps.

    
Each prompt's output feeds the next. More reliable than one giant prompt because every step is small enough to verify.

6 ReAct — Reason + Act

    
Chain of Thought plus tool use. The pattern behind agents.

    
The model loops through Thought → Action → Observation: it reasons, calls a tool, reads the result, and reasons again — until it can answer.

7 Tree of Thoughts

    
Explore several reasoning paths, prune the dead ends.

    
Instead of one line of reasoning, the model branches into multiple candidate paths, scores them, and backtracks from the bad ones — like searching a tree. Costlier, but stronger on hard problems where the first idea is often wrong.

8 System Prompts & Guardrails

    
The instruction layer, and the enforcement layer around it.

    
The system prompt is the hidden, top-priority instruction set before the user types. But prompt instructions are soft — a clever user can talk the model out of them. Guardrails are a separate hard layer that checks inputs and outputs independently.

    

      
Live: the two layers around the model

      
      
Input guardrails block bad requests before the model; output guardrails catch bad responses before the user.

    

    
Product territory: this soft-prompt vs hard-guardrail split is exactly why an external AI gateway / guardrails layer exists — you can't secure an LLM with prompt text alone.

  


  
  

    
9 Prompt Injection & Jailbreaks ★ #1 risk

    
The dark side of "the model treats all text as instructions."

    
Prompt injection smuggles malicious instructions through data the model reads — a webpage, an email, a document. The model can't inherently tell your instructions from an attacker's; it's all just tokens. Jailbreaks are a cousin that target the model's safety training directly.

    

      
Live: a hijack — then a guardrail stops it

      

      

        
        
        
      

      

    

    
Injection ≠ jailbreak: injection targets your app's instructions (hidden in data); a jailbreak targets the model's safety rules. Agents make injection far worse — a hijacked agent can take real actions.

  


  
  

    
🎯 Context Engineering

    
The evolution of prompt engineering — managing the whole payload.

    
Prompt engineering is wording one prompt well. Context engineering is deciding what fills the entire context window: system prompt + retrieved docs (RAG) + tool outputs + history + examples — in the right order, within budget. As models get agentic, this is the real skill.
