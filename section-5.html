1 Reverse Proxy → Gateways

Every "gateway" is a specialized reverse proxy. Start from what you know.

A reverse proxy sits in front of backends: one entry point, TLS termination, load balancing, a security choke point. An API gateway adds API smarts (auth, rate limits). An AI gateway adds LLM smarts (tokens, guardrails, model routing). An MCP gateway governs agent↔tool traffic.

Live: peel back the layers of specialization
Your home turf: this is the concept ADCs have owned for 20 years. AI just adds new backends and new routing logic on top of the same reverse-proxy foundation.

2 Why LLM Traffic Is Different ★ the aha

Classic load balancing assumes short, uniform, stateless requests. LLMs are the opposite.

Every "advanced" idea later in this section exists because of this mismatch.

PropertyWeb requestLLM request
Duration~millisecondsseconds–minutes
Responseall at oncestreamed token-by-token
Cost~uniformwildly variable (tokens)
Backend statestatelessKV cache matters
Backendcheap, identicalcostly GPUs, maybe diff. models

3 Load Balancing for LLMs

Round-robin is great for web servers — and bad for LLMs.

Round-robin ignores that one request is 10 tokens and another is 10,000. Token-aware balancing routes by the work, keeping GPUs evenly loaded.

4 Model Routing

Send each request to the right model — cheap for easy, smart for hard.

A gateway can classify each request and route it: a small fast model for "what's 2+2", a big expensive one for a hard reasoning task. This one decision directly controls cost and quality.

Total cost so far: $0.000 — vs $0.000 if everything went to the big model

5 Streaming, SSE & Buffering

Why the gateway must never hold the whole response.

LLMs generate token-by-token, so output is streamed as it's produced (that "typing" effect), usually over SSE (Server-Sent Events, one-way server→client). If a gateway buffers the full response before forwarding, the user stares at a blank screen the whole time.

6 Transport: HTTP/2, gRPC, WebSockets

The protocols under the hood.

SSE rides on HTTP/2, which multiplexes many streams over one connection. gRPC (binary, fast) is used for internal service-to-service calls. WebSockets add full two-way communication for interactive cases.

7 KV Cache-Aware Routing ★ the clever one

The routing decision a normal load balancer simply cannot make.

A conversation's KV cache (Section 2) lives on a specific GPU. Route the follow-up turn to that same GPU and the cache is still warm — instant. Send it elsewhere and the backend must recompute the whole history. Only a gateway that understands LLM internals can do this.

Live: send a follow-up turn — smart vs blind

8 GPU-Aware Routing & Model Affinity

Route on real hardware telemetry, not blind rotation.

GPU-aware routing reads live VRAM headroom, utilization, and queue depth (Section 4) and sends work where the hardware can take it. Model affinity keeps requests on backends that already have the needed model loaded — avoiding a costly swap in and out of VRAM.

Live: route by live GPU telemetry

9 Rate Limiting & Retries

Count tokens, not requests. Retry smartly, not blindly.

Rate limiting for LLMs is measured in tokens per minute (and cost per tenant) — one request can consume 100× another, so requests/sec is meaningless. Retries need backoff and failover to another model/provider, without triggering a retry storm.

🎯 AI Traffic Management

All of it together — the umbrella, and the whole point.

An AI-aware control plane that understands tokens, KV caches, GPUs, models, and streams — routing, protecting, and optimizing AI traffic end to end. It's a direct evolution of what an ADC already does for web and app traffic.

The one-liner, visualized
An AI gateway is a reverse proxy
that understands what's happening inside the LLM —
tokens · KV cache · GPUs · models · streams
— and routes accordingly.
Test yourself: can you name three routing decisions a normal load balancer can't make, and why? Then you've hit the goal. ✅
